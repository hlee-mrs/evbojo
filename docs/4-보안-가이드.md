# 🛡️ 보안·과부하 방어 가이드

## 현재 상태 요약
| 구간 | 방어 상태 |
|---|---|
| **지금 (GitHub Pages)** | GitHub 글로벌 CDN이 DDoS·트래픽 폭주를 대신 흡수 — 별도 조치 불필요 ✅ |
| **NAS 이전 후** | 아래 3중 방어 적용: ① Cloudflare(무료) ② nginx 자체 방어(적용 완료) ③ NAS/공유기 설정 |

## 1층 — Cloudflare (가장 중요, 무료)
집 회선은 대역폭이 작아서 **NAS 앞단의 Cloudflare가 사실상 DDoS 방어의 전부**입니다.
도메인 가이드(2번 문서)의 Cloudflare 연결을 마친 뒤 대시보드에서:

1. **Proxy(주황 구름) ON** — 집 IP가 숨겨져 직접 타격 자체가 불가능해짐 (가장 중요)
2. Security → Settings → **Bot Fight Mode ON** (무료)
3. Security → WAF → Managed rules **ON** (무료 기본 룰셋)
4. Security → **Rate limiting rule 1개 생성** (무료 플랜 1개 제공):
   - If URI Path contains `/` and Rate > `120 req / 1 min per IP` → Block 10분
5. Caching → Cache Rules: `/data/*` → Cache Eligible, Edge TTL 10분
   - 트래픽 폭주 시에도 원본(NAS)엔 10분에 1번만 도달
   - `/assets/*` → Edge TTL 7일
6. Caching → **Always Online ON** — NAS가 꺼져도 캐시로 서빙
7. SSL/TLS → **Full** 모드 + Edge Certificates → Always Use HTTPS ON
8. (공격 당하는 중일 때) Overview → **Under Attack Mode** 토글 — 방문자에게 5초 JS 검문

## 2층 — nginx 자체 방어 (적용 완료 ✅)
`deploy/nginx.conf` + `deploy/security.conf`에 이미 반영되어 있습니다:

| 항목 | 설정 | 효과 |
|---|---|---|
| 요청 속도 제한 | IP당 10 req/s, 버스트 30, 초과 시 429 | 스크래퍼·소규모 플러딩 차단 (사람은 안 걸림) |
| 동시 연결 제한 | IP당 20 | 연결 고갈 공격 방지 |
| 타임아웃 단축 | header/body/send 10s | Slowloris(느린 연결 점유) 방어 |
| 메서드 제한 | GET/HEAD만 허용 | 정적 사이트에 불필요한 공격면 제거 |
| 스캐너 차단 | sqlmap·nikto 등 UA 403, /wp-*·*.php 404 | 자동 취약점 스캔 소음 제거 |
| 보안 헤더 | CSP·nosniff·frame·referrer·permissions | XSS·클릭재킹·스니핑 방어 |
| 리소스 상한 | web 256MB/1CPU, updater 1GB/1.5CPU | 폭주해도 NAS 본체 보호 |
| 429 전용 페이지 | site/429.html | 차단당한 사용자에게 친절한 안내 |

적용/재적용: `docker compose -f deploy/docker-compose.yml up -d --force-recreate web`
확인: `docker exec evbojo-web nginx -t`

## 3층 — NAS·공유기
- [ ] 포트는 **80/443만** 포워딩 (DSM 관리포트 5000/5001은 절대 외부 개방 금지)
- [ ] Cloudflare 사용 시: 공유기에서 80/443 접근을 **Cloudflare IP 대역만 허용**하면 우회 직접공격도 차단 (Synology 방화벽 → 허용 IP에 cloudflare.com/ips 대역 등록)
- [ ] DSM 제어판 → 보안 → **자동 차단 ON** (로그인 5회 실패 시 IP 차단), 2단계 인증 ON
- [ ] SSH는 사용할 때만 켜기, 기본 admin 계정 비활성화
- [ ] Container Manager 이미지 자동 업데이트 알림 ON (nginx 보안 패치)

## 데이터·운영 보안
- 사이트는 **개인정보를 수집하지 않는 정적 사이트**라 유출될 DB 자체가 없음 (공격 가치 낮음)
- updater는 아웃바운드(ev.or.kr)만 하고 인바운드 포트가 없음
- 백업: `site/data/*.json.bak` 자동 + Hyper Backup 주간 권장
- 이상 징후 확인: `docker logs evbojo-web | grep " 429 "` (차단 발생 내역), Cloudflare Analytics → Security 탭

## 부하 한계 참고치
정적 파일 + gzip + Cloudflare 캐시 조합이면 **월 100만 PV도 NAS 부하는 미미**합니다
(원본 도달 트래픽은 캐시 미스분뿐 — 일반적으로 전체의 5~10%).
